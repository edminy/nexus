# 前端设计规范

## 1. 文档目标

本文档只保留当前仍然有效的前端约束，不再保留大段抽象设计宣言。

核心目标：

- `launcher` 和 `/app` 视觉角色分离
- 消息区、工作区、侧边栏收敛到同一套轻表面语言
- UI 先保证信息结构，再谈装饰

## 2. 页面角色

### 2.1 `launcher`

- 负责启动协作
- 允许更强的入口视觉
- 只保留一个主要焦点

### 2.2 `/app`

- 负责连续工作
- 强调工具感和信息密度
- 不做营销页式构图

## 3. `/app` 的固定规则

### 3.1 层级

最多四层：

1. 背景
2. 侧边 rail
3. 主内容 plane
4. 浮层

### 3.2 表面语法

主语法只保留：

- rail
- plane
- card

### 3.3 状态与动作分离

- 状态：文字 / 轻 badge / 状态点
- 动作：按钮 / 菜单 / 链接

## 4. 消息区规则

### 4.1 视觉体系

代码块、表格、工具结果、权限块、AskUserQuestion 必须属于同一套轻表面系统。

### 4.2 时间线

- 圆点和轨道服务于结构，不抢内容
- 对齐以真实内容锚点为准，不靠硬编码 top
- 用户消息先于本轮 Agent 回复；已完成回复按公区发布时间展示，活动 Agent 状态统一收在已完成回复之后
- 快 Agent 不等待慢 Agent；慢 Agent 只保留一个紧凑状态卡，最终回复到达后替换该状态
- guide、queue、wake 等控制信息不做成完整消息气泡，附着在目标卡片或 Thread

### 4.3 消息密度

- 同一正文只出现一次；引导消息不在全局位置和目标卡片重复渲染
- `@Agent` 使用小头像 + 可点击名称，头像是身份提示，不扩展成独立卡片
- 主 Feed 只展示正文、最终状态和一行摘要；过程细节、完整引导链和队列审计进入 Thread
- 状态变化更新原位置，不因 stream、guide 或 queue 事件制造新的视觉层级

### 4.4 文本

- 用户消息和结果消息都支持 Markdown
- 中文正文统一走消息字体链

## 5. 工作区规则

- 编辑区和文件树是工具界面，不做厚卡片
- 分栏拖拽手柄统一成轻分割线语法
- 文件树密度可高，但必须可读

## 6. 侧边栏规则

- 侧边栏主要承担导航和摘要，不应承载重数据面板
- 头像、箭头、分组标题必须共用同一中心线
- 能力侧边栏优先显示摘要，而不是全量数据

## 7. 动效规则

每页最多 2 到 3 类动效：

- 入场
- 悬浮 / reveal
- 局部过渡

超过这个预算就是噪音。

## 8. 组件实现规则

- 不为了“好看”引入额外包裹层
- 组件命名和目录要贴近职责
- shared/ui 只放真正共享的 UI，不混业务容器

## 9. 禁止项

- 同页出现多套表面语言
- 小控件过度 glass 化
- 为装饰牺牲可读性
- 通过增加壳体解决层级问题

## 10. 一句话总结

Nexus 前端应该像一套克制的协作工具，而不是一组视觉效果的集合。
