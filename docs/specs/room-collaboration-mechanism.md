# Room 协作机制

这份文档用于编写 `scope: room` 的 Skill。它说明 Room 里多个 Agent 应该怎样协作，以及写 Skill 时需要约定哪些行为边界。

## 适用场景

Room Skill 适合定义多人协作规则，例如：

- 多个 Agent 分工处理同一件事。
- 一个 Agent 负责主持、分派、汇总或复核。
- 某些信息需要先私下收集，再由指定成员公开推进。
- 需要约定谁在什么情况下发言、沉默、转交或收口。

如果 Skill 只影响单个 Agent 的行为，优先写普通 Agent Skill，不要写 Room Skill。

## 基本原则

- 公开消息会被 Room 内所有成员看到，适合宣布进展、分派任务、汇总结论和请求公开回应。
- 私下消息只给指定成员看，适合补充背景、征求意见、交接中间结果或记录不需要公开的上下文。
- 只有需要某个成员行动时才 `@成员`；说明计划、列流程、总结状态时不要随手 `@`。
- 被请求的成员如果没有需要补充的内容，应明确保持沉默或简短说明无需处理，不要制造无意义回复。
- 私下收到的信息不能自动公开。需要公开时，应只公开必要结论，不要原样转述敏感上下文。
- 协作规则应该让人看得懂：谁负责发起、谁负责响应、谁负责汇总、什么时候结束。

## 可用工具

Room 中常用的专用协作工具有两个。普通公开回复不需要调用工具，直接回复即可。

| 工具 | 用途 | 什么时候用 |
| --- | --- | --- |
| `send_directed_message` | 给指定成员发送私下消息 | 私下补充背景、点名某个成员处理、小范围收集意见、记录不需要公开的信息 |
| `publish_public_message` | 主动发布一条公开消息 | 当前在私下流程里，但需要把必要结论或进展同步给全体成员 |

有些运行环境会把工具显示成完整名称，例如 `mcp__nexus_room__send_directed_message` 和 `mcp__nexus_room__publish_public_message`。写 Skill 时可以写短名，同时说明它们是 Room 工具。

### `send_directed_message`

用于私下发送消息。常用参数：

- `recipients`：接收成员的 `agent_id` 列表。
- `content`：私下消息正文。
- `wake_policy`：是否让接收成员处理这条消息。
- `delay_seconds`：延迟处理时使用。
- `reply_route`：接收成员处理后，回复应该去哪里。
- `correlation_id`：可选，用于把一组协作动作关联起来。

常见写法：

| 目的 | 推荐参数 |
| --- | --- |
| 只做私下记录，不打扰对方 | `wake_policy: "none"`，`reply_route: { "mode": "none" }` |
| 请某个成员公开回答 | `wake_policy: "immediate"`，`reply_route: { "mode": "public" }` |
| 请某个成员私下回给主持人 | `wake_policy: "immediate"`，`reply_route: { "mode": "private", "recipients": ["主持人agent_id"], "wake_policy": "immediate" }` |
| 让多个成员先私下提供意见 | `recipients` 填多个成员，`reply_route` 按是否需要公开或回给主持人来定 |

### `publish_public_message`

用于主动发布公开消息。常用参数：

- `content`：公开消息正文。
- `correlation_id`：可选，用于把一组协作动作关联起来。

注意：

- 普通公开发言不要调用这个工具，直接回复即可。
- 只有在私下流程、工具流程或汇总流程中，需要额外把信息同步给全体成员时才使用。
- 公开消息里只有真正需要对方行动时才写 `@成员`。

## 写 Room Skill 时要说明什么

建议在 `SKILL.md` 中按下面结构写：

### 1. 这个 Skill 什么时候启用

说明适用任务、触发条件和不适用场景。

### 2. Room 成员如何分工

写清楚每类成员的职责，例如主持、执行、审核、补充信息、做最终总结等。

### 3. 公开协作规则

说明哪些内容应该发到公开区：

- 任务开始和目标确认。
- 明确分派给某个成员的行动请求。
- 阶段性进展和最终结论。
- 需要所有成员共同看到的风险、阻塞和决策。

同时说明哪些情况不应该公开发言，例如未整理草稿、未确认判断、重复确认和无意义附和。

### 4. 私下协作规则

说明哪些内容应该私下发送：

- 给某个成员的背景补充。
- 只需要指定成员处理的请求。
- 不适合公开的中间意见。
- 需要先收集再汇总的信息。

还要说明私下信息如何回到公开区：由谁判断、公开到什么程度、是否只公开结论。

### 5. 交接和收口规则

说明什么时候把任务交给别人、什么时候回到主持人、什么时候结束本轮协作。

好的收口规则通常包含：

- 当前任务是否已经完成。
- 还有谁需要继续行动。
- 最终结论应该由谁公开。
- 不需要继续回复时如何停止。

## 推荐写法

可以把下面这段交给 Agent，让它帮你生成 Room Skill：

```md
请根据这份 Room 协作机制，编写一个 `scope: room` 的 `SKILL.md`。

要求：
- 只写这个 Room Skill 的协作规则。
- 说明适用场景、成员分工、公开发言规则、私下协作规则、工具使用规则、交接和收口规则。
- 明确什么时候需要 @成员，什么时候不要 @成员。
- 明确什么时候用 `send_directed_message`，什么时候用 `publish_public_message`，什么时候直接公开回复。
- 私下信息转公开时，只公开必要结论。
- 内容面向实际使用者，简洁、可执行、不要堆技术细节。
```

## 检查清单

发布前确认：

- `scope: room` 已写清楚。
- 成员分工明确。
- 公开消息和私下消息的边界明确。
- `@成员` 的使用条件明确。
- Room 工具的使用条件明确。
- 私下信息不会被无意公开。
- 有明确的交接和结束规则。
- 文档短、直接、可执行。
